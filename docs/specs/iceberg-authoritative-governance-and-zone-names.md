# Spec: Iceberg-Authoritative Governance and Zone Naming Cleanup

**Status:** DRAFT
**Zone:** Infrastructure (cross-cutting)
**Primary Agent:** @staff-engineer
**Created:** 2026-04-28

## Problem Statement

Brightsmith is moving toward Iceberg as the only source of truth for governance state, but the current implementation still behaves like a dual-write system:

1. Runtime modules write Iceberg and files in the same operation.
2. Several Iceberg write failures are swallowed and followed by file writes.
3. Generated governance files can drift from Iceberg tables.
4. `governance_db.py` has grown into a monolithic module containing schemas, writers, readers, migration, export, CLI commands, and markdown parsing.
5. The codebase mixes zone names: `raw`, `base`, `consumable`, `ai_ready` alongside `bronze`, `silver`, `gold`, and `mcp`.
6. Tests cover many happy paths and idempotency cases, but do not consistently enforce failure behavior or prove that files are derived from Iceberg.
7. Some docs, comments, and tests have stale wording such as "dual-write", "7 tables", or "15 tables", which makes the implementation feel generated rather than owned.

## Decision

Iceberg is the system of record for all generated governance state.

Runtime code must write governance state only to Iceberg. Files under `governance/` that are generated by Brightsmith are derived exports only. If a file is needed for human readability, review, or compatibility, it must be rendered by reading Iceberg tables.

Governance state is split into two Iceberg-backed catalogs:

- **Enterprise governance catalog:** shared standards that should be reused across products, such as business terms, CDE identification criteria, criticality classifications, PII classifications, policy definitions, standard DQ rule templates, allowed value sets, and stewardship ownership.
- **Product governance catalog:** product-specific runtime and implementation state, such as specs, contracts, contract-to-term mappings, DQ runs, approvals, CAB decisions, lineage, exceptions, and scorecards.

Product governance artifacts reference enterprise governance artifacts by stable IDs. They must not copy enterprise definitions except when producing an export snapshot.

The canonical zone names are:

| Canonical | Deprecated Alias |
|-----------|------------------|
| `bronze` | `raw` |
| `silver` | `base` |
| `gold` | `consumable` |
| `mcp` | `ai_ready` |

Deprecated aliases may be accepted at boundaries during migration, but persisted state, docs, generated artifacts, tests, and user-facing text must use canonical names.

## Goals

- Make Iceberg the only runtime write target for generated governance state.
- Separate reusable enterprise governance standards from product-specific governance/runtime state.
- Make product governance records reference enterprise governance records by stable IDs instead of duplicating definitions.
- Replace dual-write behavior with explicit export commands that read from Iceberg.
- Fail loudly when required Iceberg writes fail.
- Split governance storage code into maintainable modules.
- Standardize zone names to `bronze`, `silver`, `gold`, and `mcp`.
- Add tests that enforce the new source-of-truth contract.
- Clean stale comments, docstrings, test names, and lint issues introduced by the governance expansion.

## Non-Goals

- Do not remove human-authored design specs under `docs/specs/`.
- Do not remove session logs under `docs/sessions/`.
- Do not remove non-governance bootstrap files such as project manifests or source connection configs unless a separate project-configuration spec replaces them.
- Do not redesign the Iceberg table schemas unless required to remove file dependency or fix a correctness issue.
- Do not change the medallion architecture beyond naming standardization.

## Runtime Contract

### Required behavior

Runtime actions write generated governance state to Iceberg:

```text
runtime action
  -> validate input
  -> write Iceberg table rows
  -> return structured result
```

File generation is a separate export path:

```text
export command
  -> read Iceberg table rows
  -> render markdown/json/yaml files
  -> report generated paths
```

### Prohibited behavior

Runtime code must not:

- write a generated governance file after writing Iceberg
- use a generated governance file as fallback source of truth
- swallow required Iceberg write errors and continue as if the operation succeeded
- update a file and then call a sync function to backfill Iceberg

### Failure behavior

Required Iceberg writes must raise exceptions or return explicit failure status that callers cannot ignore.

Allowed exception: best-effort telemetry can remain fault-tolerant only if it is not part of the governance state contract and is documented as such. The default should be fail-loud.

## File Classification

### Generated governance state

These should be written only to Iceberg at runtime and generated as exports when files are needed:

- DQ rules
- DQ results
- DQ acknowledgments
- DQ scorecards
- pipeline events and state
- contract metadata and generated contracts
- contract column metadata
- CAB decisions
- golden datasets
- run history
- chaos manifests
- reviews
- approval records
- audit trail entries
- data dictionaries
- model entities, columns, and relationships
- policies generated by agents
- lineage documents generated from runtime lineage

### Non-governance files

These may remain files because they are not governance state:

- `docs/specs/*.md`
- `docs/sessions/*.md`
- `domain/manifest.yaml`
- `domain/sources/*.yaml`

Human-entered governance inputs must not remain file-authoritative. Business glossaries, policies, DQ rules, approvals, collision/normalization rules, CDE/PII tags, data contracts, golden datasets, and governance reviews are governance state. They must be created or edited through Iceberg-backed APIs and exported to files only when needed for human readability or compatibility.

## Governance Catalog Split

### Enterprise governance catalog

The enterprise catalog contains reusable standards that apply across products. It is the shared vocabulary and policy layer.

Recommended namespace:

```text
governance_enterprise
```

Recommended tables:

| Table | Purpose |
|-------|---------|
| `governance_enterprise.business_terms` | Canonical business terms and definitions |
| `governance_enterprise.cde_criteria` | Standard criteria for deciding whether a product attribute is a CDE |
| `governance_enterprise.criticality_classifications` | Standard criticality levels and usage guidance |
| `governance_enterprise.pii_classifications` | Standard PII and sensitivity classifications |
| `governance_enterprise.policies` | Enterprise policy definitions |
| `governance_enterprise.dq_rule_templates` | Standard reusable DQ rules and applicability metadata |
| `governance_enterprise.allowed_values` | Reusable enumerations and reference value sets |
| `governance_enterprise.data_domains` | Enterprise data domains and domain ownership |
| `governance_enterprise.stewards` | Stewardship ownership and approval groups |

Enterprise records require stable IDs and versions. Example IDs:

```text
BT-REVENUE-001
CDECRIT-REGULATORY-REPORTING
CRIT-HIGH
POL-RETENTION-FINANCIAL-001
PII-TAX-ID
DQR-NOT-NULL-CDE
```

Enterprise records are centrally governed. Product workflows may propose new enterprise terms or policy updates, but accepted changes are promoted through an enterprise approval path.

### Product governance catalog

The product catalog contains product-specific implementation and runtime state.

Recommended namespace:

```text
governance_product
```

Recommended tables:

| Table | Purpose |
|-------|---------|
| `governance_product.spec_registry` | Product specs and lifecycle state |
| `governance_product.contract_metadata` | Product contract metadata |
| `governance_product.contract_columns` | Product column metadata and enterprise term references |
| `governance_product.dq_rules` | Product-specific rule instances, including references to enterprise templates |
| `governance_product.dq_runs` | Runtime DQ execution summaries |
| `governance_product.dq_rule_results` | Runtime DQ rule outcomes |
| `governance_product.dq_acknowledgments` | Product-specific acknowledgments/exceptions |
| `governance_product.pipeline_events` | Product pipeline events and approvals |
| `governance_product.cab_decisions` | Product schema change decisions |
| `governance_product.golden_datasets` | Product-specific verification values |
| `governance_product.lineage_events` | Product lineage events and derived lineage docs |
| `governance_product.documents` | Product reviews, audits, and generated docs |

Product records should reference enterprise records by ID:

| Product Field | Enterprise Reference |
|---------------|----------------------|
| `business_term_id` | `governance_enterprise.business_terms.term_id` |
| `cde_criteria_ids` | `governance_enterprise.cde_criteria.criteria_id` |
| `criticality_classification_id` | `governance_enterprise.criticality_classifications.classification_id` |
| `policy_ids` | `governance_enterprise.policies.policy_id` |
| `pii_classification_id` | `governance_enterprise.pii_classifications.classification_id` |
| `dq_template_id` | `governance_enterprise.dq_rule_templates.template_id` |
| `allowed_value_set_id` | `governance_enterprise.allowed_values.value_set_id` |

### Reference, do not duplicate

Product artifacts must not duplicate enterprise definitions. CDE is not a reusable semantic entity and should not be modeled as a separate `cde_id`. CDE status is a product attribute classification: the same business term can be critical in one product/table and non-critical in another.

For example, product contract columns should store:

```text
column_name = acct_num
business_term_id = BT-ACCOUNT-NUMBER-001
is_cde = true
cde_rationale = "Used in customer exposure reporting submitted to regulators"
cde_criteria_ids = ["CDECRIT-REGULATORY-REPORTING"]
criticality_classification_id = CRIT-HIGH
policy_ids = ["POL-CUSTOMER-DATA-RETENTION-001"]
```

A different product column can map to the same business term without being a CDE:

```text
column_name = acct_num
business_term_id = BT-ACCOUNT-NUMBER-001
is_cde = false
cde_rationale = ""
criticality_classification_id = CRIT-LOW
```

For a financial column:

```text
column_name = revenue
business_term_id = BT-REVENUE-001
is_cde = true
cde_rationale = "Feeds audited financial reporting package"
cde_criteria_ids = ["CDECRIT-EXTERNAL-FINANCIAL-REPORTING"]
policy_ids = ["POL-RETENTION-FINANCIAL-001"]
```

They should not copy the full definition of revenue, policy text, CDE criteria text, or stewardship metadata into the product table. Query views and exports may join enterprise and product catalogs to produce denormalized snapshots for human review.

### Enterprise reuse queries

The split must support enterprise-level discovery:

- Which products use business term `BT-REVENUE-001`?
- Which product columns are flagged `is_cde = true` and why?
- Which products use CDE criterion `CDECRIT-REGULATORY-REPORTING`?
- Which product columns are governed by policy `POL-RETENTION-FINANCIAL-001`?
- Which DQ rule templates are reused most often?
- Which products have local exceptions against an enterprise standard?

## Zone Naming Standard

### Canonical names

Use canonical medallion names everywhere:

- `bronze`: raw ingested/source-shaped tables
- `silver`: cleaned, conformed, normalized tables
- `gold`: business-ready data products
- `mcp`: AI-serving layer and MCP tools

### Deprecated aliases

The aliases `raw`, `base`, `consumable`, and `ai_ready` are deprecated.

During migration, boundary adapters may accept aliases and normalize them:

```python
ZONE_ALIASES = {
    "raw": "bronze",
    "base": "silver",
    "consumable": "gold",
    "ai_ready": "mcp",
}
```

Persisted values must be canonical. If an alias is received, normalize before writing Iceberg.

### Search and replace targets

Rename user-facing and persisted references in:

- `src/brightsmith/**/*.py`
- `tests/**/*.py`
- `docs/**/*.md`
- `agents/**/*.md`
- `.claude/agents/**/*.md`
- `skills/**/*.md`
- `governance/**/*.md`
- example configs under `domain/`

Do not blindly replace unrelated English prose. For example, "raw data" may be valid prose, but a zone label, table namespace, path, test fixture, command output, or contract value should use `bronze`.

## Target Architecture

Split the governance storage layer into focused modules:

| Module | Responsibility |
|--------|----------------|
| `src/brightsmith/infra/governance/__init__.py` | Public API re-exports |
| `src/brightsmith/infra/governance/schemas.py` | Enterprise and product Iceberg schemas and table config |
| `src/brightsmith/infra/governance/repository.py` | Catalog/table access, write, query, grain ID handling |
| `src/brightsmith/infra/governance/serializers.py` | JSON, timestamp, and zone normalization helpers |
| `src/brightsmith/infra/governance/enterprise.py` | Enterprise governance write/read APIs |
| `src/brightsmith/infra/governance/product.py` | Product governance write/read APIs |
| `src/brightsmith/infra/governance/resolution.py` | Enterprise ID resolution, validation, and join helpers |
| `src/brightsmith/infra/governance/exporters.py` | File renderers that read Iceberg |
| `src/brightsmith/infra/governance/migration.py` | One-time file-to-Iceberg backfill |
| `src/brightsmith/infra/governance/parsers.py` | Legacy markdown/json parsers used only by migration |
| `src/brightsmith/infra/governance/cli.py` | CLI commands |

Keep `src/brightsmith/infra/governance_db.py` as a compatibility shim during migration:

```python
from brightsmith.infra.governance.enterprise import *
from brightsmith.infra.governance.product import *
```

After downstream imports are migrated, remove the shim in a later cleanup spec.

## Implementation Plan

### Phase 1: Enforce Iceberg-only runtime writes

Modify runtime modules so they no longer write generated governance files:

| File | Change |
|------|--------|
| `src/brightsmith/infra/contract.py` | `save_contract()` writes contract metadata/columns to Iceberg only. If YAML export is still needed, move it to an exporter. |
| `src/brightsmith/infra/dq_runner.py` | `run_rules()` writes `dq_runs` and `dq_rule_results` to Iceberg only. Remove `_save_results()` from runtime path. |
| `src/brightsmith/infra/dq_runner.py` | `acknowledge_failures()` writes `dq_acknowledgments` to Iceberg only. No ack JSON runtime write. |
| `src/brightsmith/infra/dq_scorecard.py` | Scorecard generation reads Iceberg scorecard data and renders export output only when called as export/reporting behavior. |
| `src/brightsmith/infra/pipeline_gate.py` | Pipeline events and approvals write to Iceberg only. Any JSON pipeline state file becomes export-only or compatibility-only during migration. |
| `src/brightsmith/infra/governance_db.py` | Remove runtime assumptions that files are primary, secondary, or dual-written. |

Required Iceberg writes must not be wrapped in broad `except Exception` blocks unless the exception is re-raised.

### Phase 2: Introduce enterprise/product catalog separation

Add enterprise and product governance catalog configuration.

Implementation requirements:

- Create or configure separate Iceberg namespaces for enterprise and product governance.
- Move shared artifacts such as business terms, policies, CDE criteria, criticality classifications, PII classifications, and DQ templates into the enterprise namespace.
- Keep runtime artifacts such as DQ runs, approvals, lineage, CAB decisions, and product contracts in the product namespace.
- Add validation that product records reference existing enterprise IDs where required.
- Add validation that CDE status is stored as `is_cde` on product attributes and is not inferred from `business_term_id`.
- Add query helpers that join product records to enterprise definitions for UI/export use cases.
- Add migration logic that deduplicates repeated business terms or policies across products into enterprise records, then rewrites product records to references.

### Phase 3: Add explicit exporters

Add exporter functions that read Iceberg and generate files:

| Exporter | Output |
|----------|--------|
| `export_contracts_to_files()` | `governance/data-contracts/*.yaml` |
| `export_dq_results_to_files()` | `governance/dq-results/*.json` |
| `export_dq_scorecards_to_files()` | `governance/dq-scorecards/*.md` |
| `export_pipeline_state_to_files()` | `governance/pipeline-state/*.json` |
| `export_cab_decisions_to_files()` | `governance/cab-decisions/*.json` |
| `export_documents_to_files()` | reviews, audit trails, approvals, models, policies |
| `export_all_governance_files()` | calls all exporters |

Exporter tests must prove files are generated from Iceberg by writing Iceberg rows first, then exporting, then comparing file content to queried rows.

### Phase 4: Split governance module

Move code out of `governance_db.py` into the target architecture modules.

Refactor without behavior changes first:

1. Move schemas and `_TABLE_CONFIGS`.
2. Move table access and query helpers.
3. Move write APIs.
4. Move read APIs.
5. Move migration and parsing helpers.
6. Move CLI code.
7. Keep compatibility imports in `governance_db.py`.

### Phase 5: Normalize zone names

Add a small zone normalization helper:

```python
def normalize_zone(zone: str) -> str:
    return ZONE_ALIASES.get(zone, zone)
```

Use it before:

- writing table namespaces
- writing `spec_registry.zone`
- parsing contracts
- parsing pipeline state
- generating export paths
- comparing zone names in tests

Update examples, docs, tests, and agent instructions to use canonical names.

### Phase 6: Remove stale dual-write language and lint debt

Clean:

- comments that say "dual-write"
- comments that say files are secondary runtime writes
- stale table counts
- test names that do not match assertions
- unused imports and variables in changed files

Run:

```bash
uv run python -m pytest
uv run ruff check src/brightsmith/infra/governance_db.py src/brightsmith/infra/contract.py src/brightsmith/infra/dq_runner.py src/brightsmith/infra/dq_scorecard.py src/brightsmith/infra/pipeline_gate.py tests/infra/test_governance_db.py
```

The touched-file ruff check must pass. Full-repo lint cleanup can be separate if pre-existing issues are out of scope.

## Testing Plan

### Source-of-truth tests

- Runtime DQ run writes Iceberg rows and creates no `governance/dq-results/*.json`.
- DQ acknowledgment writes Iceberg row and creates no ack JSON file.
- Contract save writes Iceberg metadata/columns and creates no contract YAML file in runtime path.
- Pipeline gate step writes Iceberg event and creates no pipeline JSON file in runtime path, after migration of compatibility behavior.
- Scorecard export reads Iceberg and writes markdown only through exporter.

### Failure-mode tests

Patch Iceberg write functions to raise and assert the caller fails:

- `run_rules()` fails when `write_dq_run()` fails.
- `save_contract()` fails when `sync_contract()` fails.
- `acknowledge_failures()` fails when `write_dq_acknowledgment()` fails.
- pipeline event write failure is visible to the caller.

### Export tests

- Write governance rows directly to Iceberg.
- Run each exporter.
- Assert generated file content matches Iceberg state.
- Delete generated files and rerun exporter to prove reproducibility.

### Enterprise reuse tests

- Product contract columns can reference enterprise `business_term_id`, `cde_criteria_ids`, `criticality_classification_id`, `policy_ids`, and `dq_template_id`.
- Product contract columns store `is_cde` as an independent boolean classification.
- Two columns with the same `business_term_id` can have different `is_cde` values.
- A glossary/business-term record with any legacy CDE-like fields must not set or override product-column `is_cde`.
- Writing a product record with an unknown required enterprise ID fails validation.
- A query can list all product columns using a given business term ID.
- A query can list all product columns where `is_cde = true`.
- A query can list all products governed by a given policy ID.
- Exports may include denormalized enterprise definitions, but persisted product records store only stable references.

### Zone naming tests

- Aliases normalize before persistence: `raw -> bronze`, `base -> silver`, `consumable -> gold`, `ai_ready -> mcp`.
- Persisted `spec_registry.zone` values are canonical.
- Contract namespaces are canonical.
- DQ table references and examples use canonical namespaces.
- Docs/tests do not contain deprecated zone names except in migration/alias documentation.

### Regression tests

- Existing governance query APIs return the same logical records after module split.
- Existing imports from `brightsmith.infra.governance_db` continue to work during compatibility period.
- `uv run python -m pytest` passes.

- [ ] Enterprise governance artifacts live in a shared enterprise governance namespace.
- [ ] Product/runtime governance artifacts live in a product governance namespace.

- [ ] Migration/backfill code is separate from runtime write/query code.
- [ ] Markdown and legacy file parsers are used only by migration/import paths.
- [ ] Canonical zone names are `bronze`, `silver`, `gold`, and `mcp`.

## Migration Notes

Existing generated files should be treated as legacy import data:

1. Run one-time migration from files into Iceberg.
2. Validate row counts and spot-check content.
3. Archive or delete generated runtime governance files.
4. Use exporters to regenerate any files still needed for human review.

After migration, file-to-Iceberg sync must not run as part of normal runtime. It remains a recovery or import tool.

## Risks

- Some agent instructions may still tell agents to write files directly. These must be updated or they will reintroduce drift.
- Some tests may currently depend on file side effects. Those tests should move to exporter tests.
- External consumers may still read generated files. They need either Iceberg access or an explicit export step.
- Zone renaming can break fixtures or examples if aliases are replaced mechanically without normalization.

- Should the compatibility shim preserve all existing `governance_db.py` private helpers, or only public APIs?
