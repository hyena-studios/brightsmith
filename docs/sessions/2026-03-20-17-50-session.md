## Decisions Made
- **Agent is diagnostic-first, not design-first.** The old temporal-modeler assumed bitemporal was the answer and jumped to Iceberg schema patterns. The new agent asks four diagnostic questions and can recommend "no temporal modeling needed" — which is the right call for many datasets.
- **Storage-layer agnostic.** Transaction time can be infrastructure-managed (Iceberg snapshots) or schema-managed (explicit columns). The agent recommends based on what's available, not what Brightsmith happens to use.
- **Supersession grain is an explicit concept.** Learned from EDGAIR: the key insight is defining which fields constitute "the same fact across versions" vs. which fields discriminate versions. Made this a first-class output of the agent.
- **No agents should be skippable (except zone-scoping).** Entity-resolver, pii-scanner, and adversarial-auditor were all skippable with "we think we know the answer" conditions. Same anti-pattern as temporal-modeler. Made all non-skippable — the agent runs and decides "not needed" when that's the right call.
- **AI-Ready zone is a governed AI query layer, not just an MCP server.** Five components: MCP server (exists), formatter (new), anomaly checker (new), system prompt builder (new), response enrichment pipeline (new). Learned from EDGAIR building all three ad-hoc.
- **Intelligence layer is optional and composable.** Each component works independently. A domain that doesn't need formatters doesn't use them. The MCP server works fine without any intelligence layer components (current behavior preserved).
- **Framework provides mechanism, domain provides content.** BaseFormatter doesn't know what a currency is. BaseAnomalyChecker doesn't know about negative equity. BaseSystemPrompt doesn't know about fiscal years. Domain projects register format rules, anomaly rules, and prompt sections.
- **Lineage: runtime emission exists, everything downstream is missing.** Brightsmith has solid emit_start/emit_complete/emit_fail to Iceberg. But no CLI to query events, no governance doc generation, no column-level lineage, no contract integration, no MCP tool fix. All framework-level gaps, not domain-specific.
- **Column-level lineage is structural, not runtime.** Stored in governance JSON docs, not in the Iceberg events table. Doesn't change per execution. `generate-docs` merges structural (column mappings) with operational (runtime metrics).
- **Slug-based doc naming over hardcoded mapping.** EDGAIR had a hardcoded job-to-file dict. Framework uses `job_name.replace(".", "-") + ".json"` — no framework changes per domain.
- **Auto-populate contract lineage.** If lineage events and contracts exist for the same table, the relationship is deterministic. Manual entry creates drift risk.
