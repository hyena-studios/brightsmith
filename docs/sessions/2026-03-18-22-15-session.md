## Decisions Made
1. **Change 9 added to spec before implementation** — user wanted programmatic gate enforcement, so we designed it collaboratively and added it to the spec before building. This keeps the spec as source of truth.
2. **PipelineGate as a single Python module** (not a package) — matches the pattern of dq_runner.py, period_disambiguator.py. Simple enough for one file.
3. **Step definitions as frozen dataclasses in tuples** — immutable, hashable, can't be accidentally modified at runtime. Tuples (not lists) reinforce immutability.
4. **SHA-256 tamper evidence on output files** — hashes recorded at completion time, audit report can detect post-completion modifications. Not enforcement (modifications may be legitimate), but detection.
5. **Three recording locations for approvals** — pipeline state (machine-readable), audit trail (governance artifact), session log (chronological). Redundancy is intentional for audit defensibility.
6. **Consumable zone steps reuse base greenfield/backfill definitions** — same structural pattern. If they diverge later, easy to split.
