## Decisions Made
- All governance DB writes are fault-tolerant (try/except with logging) — if the Iceberg write fails, the file-based workflow continues unimpaired. This follows the lineage.py pattern.
- Query function uses DuckDB `rel.description` before `fetchall()` to get column names (DuckDB sets `con.description = None` after fetchall).
- `_write_records()` converts timestamp grain fields to ISO strings before hashing, so grain IDs are deterministic regardless of timestamp object type.
- Grain ID prefixes are derived from first 4 chars of table name (e.g., SPEC, DQ_R, PIPE, CONT, GLOS, AGEN).
- The `sync_from_files()` function extracts table names from DQ rules files when building DQ run records, since results files don't store table names directly.

## Problems Encountered
- DuckDB `con.description` is `None` after `fetchall()` — had to restructure query to capture description from the relation object before consuming rows.
