## Governance Review: governance-database-only
**Review Type:** Pre-Implementation
**Reviewer:** @governance-reviewer
**Date:** 2026-03-29
**Verdict:** CHANGES REQUESTED

### Summary

This spec proposes migrating all file-based governance artifacts to Iceberg tables, making the database the sole source of truth. The ambition is correct and the motivation is well-articulated. However, the spec has significant gaps in coverage, schema design concerns, high-risk migration decisions that need safeguards, and scope issues that must be addressed before implementation.

### Decision Rationale

**Verdict: CHANGES REQUESTED** -- The spec's direction is correct and the five design decisions are generally sound. However, implementation cannot proceed because:

1. **Coverage gaps are blocking.** At least 6 file-writing/reading modules are not addressed (lineage generate-docs, glossary validator/loader, staging, EDA reports, collision rules, data dictionary). Implementing without these creates a half-migrated state that is WORSE than the current dual-write pattern, because some artifacts would be Iceberg-only while others remain file-only, with no clear documentation of which is which.

2. **Schema design needs clarification.** The `dq_rules` grain including `status` and the `golden_datasets` grain including a JSON string both create ambiguity in the dedup/query patterns. These must be resolved before implementation to avoid data correctness issues.

3. **Migration risk is underestimated.** A single-pass migration of 10+ modules with no rollback plan, no migration validation, and no fallback CLI is risky for a governance system. Governance data is the metadata that tells you whether your pipeline is healthy -- if governance itself breaks, you are flying blind.

4. **Documentation scope is not enumerated.** CLAUDE.md and 15+ agent definitions reference file paths that will become invalid. Without enumerating these changes, the implementation will produce a working codebase with stale documentation, causing every agent to reference non-existent file paths.

### Required Changes Before Implementation

1. Add all missing modules to scope (Issues #1-7, #15-16)
2. Clarify `dq_rules` grain and read pattern (Issue #8)
3. Fix `golden_datasets` grain determinism (Issue #9)
4. Add migration rollback plan or split into phases (Issue #10)
5. Add migration validation step (Issue #11)
6. Enumerate CLAUDE.md and agent definition changes (Issue #12)
7. Address orphaned file path columns (Issues #18, #20)
8. Add CLI fallback/recovery documentation (Issue #17)

### Recommendations (Non-Blocking)

1. **Consider splitting this spec.** The spec itself estimates 9 sub-tasks. A spec touching 10+ modules, 14 Iceberg tables, 15+ agent definitions, and CLAUDE.md is unusually large. Consider: Spec A = new Iceberg tables + write functions + migration. Spec B = switch all readers to Iceberg. Spec C = documentation updates + cleanup. This reduces blast radius per spec.

2. **Reassign primary agent** from @governance-reviewer to an implementation-focused agent (Issue #14).

3. **Add an explicit "files that stay" inventory** -- the spec lists what stays but does not cross-reference against `setup.py`'s directory creation list (line 28-39), which creates 10+ governance subdirectories. After migration, `setup.py` should stop creating directories that are no longer used.

4. **Consider keeping `business-glossary.json` as a file.** The glossary is human-edited, version-controlled, and validated by `glossary_validator.py`. Unlike runtime artifacts (DQ results, pipeline state), the glossary is a curated input, not a generated output. Migrating it to Iceberg removes the ability to review glossary changes in git diffs. This is a design choice, not a bug, but it deserves explicit justification.

5. **Document the Brightforge query migration.** The spec's Problem Statement says "Brightforge must parse files." After migration, Brightforge queries Iceberg. The spec should describe the Brightforge-facing API surface (what queries Brightforge will run) to ensure the Iceberg schema supports those access patterns.
